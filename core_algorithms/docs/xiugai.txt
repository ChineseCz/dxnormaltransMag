第五章 基于知识图谱与检索增强生成的平台智能助手
在第四章完成了平台底层实时预测的异构模型基座建设后，如何降低该复杂平台在电网行业内的使用门槛，实现"人机自然交互"与"故障智能诊断"，成为了平台工程落地的关键。传统的专家平台往往依赖固化的规则树，而当前孤立的大语言模型在面对垂直工业电气领域时极易产生严重的"知识幻觉"。为此，本章创新性地构建了基于KG-RAG的电力设备电磁场实时预测平台智能体助手，且围绕电抗器与变压器的电磁场预测的典型场景，验证了该平台智能助手系统。
5.1 双维度领域知识图谱设计与实现
知识图谱作为LLM智能助手的“长期结构化记忆”，其质量直接决定了下游诊断预测的严谨性。有别于传统的知识图谱构建只关注机械硬件的单一图谱，本平台不仅包含电气设备实体（如电抗器、变压器），更在底层运行着错综复杂的异构深度学习预测算法。因此，本节创新性地设计了“工业设备运维”与“算法模型排错”相融合的双维度图谱模式，并引入大语言模型指令工程实现了知识网络的自动化构建。
5.1.1 跨域双维度图谱模式约束设计
为了实现平台软硬件全生命周期的故障排查，平台采用了自顶向下的构建策略，设计了跨越物理空间与算法空间的双维度模式，也即面向设备与算法排错的双维度知识图谱模式约束（Schema）设计，如图5-1。
物理维度：该维度主要针对电气设备畸变与硬件老化问题。平台定义了设备（Device）、核心部件（Component）、电磁场异常表征（Phenomenon）、深层故障根因（Fault_Cause）与维修对策方案（Solution）五个核心实体概念。在此基础上，构建了一条清晰的因果推演链路：通过包含关系确立设备与部件的隶属，利用表征关系描述部件劣化对电磁场的影响，并进一步通过指示逻辑实现从表观异常到深层故障根因的映射。例如，针对变压器典型故障，平台建立了(局部磁场分布异常,指示,绕组匝间短路)这一典型因果三元组，并最终通过需求关系导向具体的维修方案。
软件维度：为辅助算法工程师对第四章中的高维电磁场实时预测模型进行调优，该维度将AI训练过程中的抽象概念实体化。平台定义了模型类别（Model_Type）、训练指标（Training_Metric）、算法异常（Algorithm_Issue）与超参数调优策略（Hyperparam_Tuning）的实体类别。在此基础上，构建了面向模型研发全生命周期的语义关系网络：通过评估关系确立模型与评价指标的绑定，利用呈现关系描述指标数值对算法状态的反馈。例如，针对训练波动问题，平台定义了(验证集损失剧烈震荡，缓解，降低学习率)这一典型调优三元组，从而实现从算法异常到具体调参手段的映射，为模型优化提供决策支撑。

5.1.2 基于LLM的自动化知识抽取
在5.1.1节确立了双维度图谱的模式约束后，如何从非结构化文本中精准提取符合规范的知识三元组，成为了图谱构建的关键。传统的序列标注算法在处理“漏磁场畸变”或“验证集损失震荡”等电力与算法交叉领域的专业词汇时，往往面临泛化能力不足的问题。为此，本平台引入基于通义千问（Qwen）云端LLM的指令工程技术，在零本地显存占用的前提下，实现了高保真、低成本的图谱自动化构建。
(1) 零样本（Zero-Shot）抽取的“幻觉”痛点分析
在平台研发初期，平台尝试直接采用零样本提示词引导大语言模型进行三元组抽取。即仅向模型输入文本，而不提供任何约束与输出格式规范。例如，当输入文本为“变压器铁心局部过热通常是因为硅钢片绝缘漆脱落引起的”时，Zero-Shot模式下大模型的输出往往呈现出极大的不确定性，返回结果多为自然语言段落（如：“这段话提到了变压器铁心和过热现象...”），或者凭空捏造了图谱中不存在的关系类型（如CAUSED_BY而不是平台规范的INDICATES）。这种“输出格式不可控”与“知识幻觉”导致后端的Cypher拼装服务频繁报出解析异常，完全无法正常使用。
(2) 面向双维图谱模式的少样本（Few-Shot）指令工程设计
为解决上述痛点，本平台引入了少样本思维链指令工程。平台设计了少样本Prompt模板。该模板不仅注入了特定的领域背景，还严格约束了大模型的JSON结构输出契约。核心的结构化抽取Prompt模板如表5-1。

5.2 向量数据库构建
单纯依赖5.1节构建的知识图谱虽能解决严谨的因果溯源问题，但电网行业内仍存在大量如设备运行规程等非结构化长文本手册，这类知识难以完全抽象为点边关系的图谱节点。为此，平台引入了基于FAISS的轻量级向量数据库，对工程文档进行语义索引，与5.1节的结构化图谱共同构成双轨知识底座，为后续混合检索增强提供完整的数据支撑。
5.2.1 非结构化文本切片方法
为实现对海量工程文档的高效语义检索，平台后端构建了非结构化文档的预处理流水线。首先，当平台用户在界面上传非结构化工程文档时，平台调用文档解析引擎提取纯文本内容；针对大语言模型上下文窗口的截断限制，平台引入了递归字符文本切分器（RecursiveCharacterTextSplitter），采用滑动窗口采样策略对长文本进行分段处理：设定基础切片容量为500个Token，相邻切片间的语义重叠区设定为50个Token，从而确保跨越切片边界的关键设备参数或操作规程不会被强行割裂，完整保留了如"额定电压—绕组参数—预防性试验周期"等跨行语义关联。
在完成切片后，平台对生成的文本块执行轻量级质量过滤：剔除Token数低于30的噪声短片段（如章节标题行、页眉页脚等），并对含有大量数字表格的切片进行标记，以便在后续检索阶段优先匹配语义密度较高的段落。最后，经过上述流水线处理，平台围绕电抗器与变压器两类预测场景，初步整理导入了11份运维手册与模型调参指南文档，共生成了107条有效文本切片，构成向量索引的原始语料。
5.2.2 文本向量化与向量库实例化
文本完成切片后，机器依然无法直接理解其语义。平台在后端集成了开源的中文稠密检索模型（基于BAAI/bge-base-zh预训练权重）。该模型基于预训练的Transformer编码器架构，能够将任意长度的中文文本块T映射为一个包含丰富语义特征的高维稠密向量 ，其空间维度为 维。通过该映射，语义相近的段落（如"局部短路"与"电流极值异常"）在高维向量空间中的几何距离将被显著拉近，而语义无关的段落则在空间上相互远离。在检索阶段，平台采用基于内积优化的余弦相似度进行Top-K召回，即计算查询向量与候选文档向量之间夹角余弦值，语义越相近的切片得分越高，从而筛选出最相关的Top-K个文本块作为上下文输入。
在完成全量文本块的Embedding计算后，平台调用FAISS库构建IndexFlatIP内积优化索引，将全部文本切片的768维向量矩阵一次性加载入索引。为避免每次服务重启时重复执行耗时的全量Embedding计算，平台通过faiss.write_index()将索引文件序列化至本地磁盘，并在服务启动时通过faiss.read_index()直接加载，实现了索引的持久化复用。同时，平台维护了一份与FAISS索引行号严格对齐的元数据映射表，记录每条切片对应的原始文档名称、页码区间及所属知识类别，确保检索命中后能准确溯源。向量库核心构建逻辑如表5-4所示。
表5 4 FAISS向量库构建与持久化核心代码
import faiss
import numpy as np
from sentence_transformers import SentenceTransformer
# 加载中文稠密检索模型
embedder = SentenceTransformer("BAAI/bge-base-zh")
# 批量Embedding，返回 (N, 768) 的浮点矩阵
corpus_embeddings = embedder.encode(
    text_chunks, batch_size=64, normalize_embeddings=True, show_progress_bar=True
)
corpus_embeddings = np.array(corpus_embeddings, dtype="float32")
# 构建内积优化的FAISS平坦索引
dimension = corpus_embeddings.shape[1]
index = faiss.IndexFlatIP(dimension)
index.add(corpus_embeddings)
# 持久化索引到本地磁盘
faiss.write_index(index, "faiss_kg_rag.index")
print(f"FAISS向量库构建完成，共索引 {index.ntotal} 条文本切片。")
5.3 混合检索增强架构设计
基于图数据库的结构化逻辑检索与向量库的非结构化语义检索各有优势，前者擅长精准的因果推演，后者擅长广域的语义相似匹配。单独依赖单一检索路径均存在局限性，纯图检索无法覆盖未结构化的操作规程细节，纯向量检索则难以保证因果逻辑的严谨性。为此，本节在双轨知识底座的基础上，设计了融合意图路由与并发调度的混合RAG架构，实现两条知识链路的精准协同调度。
5.3.1 结构化图轨迹检索实现
与基于语义相似度的向量检索不同，图轨迹检索侧重于对5.1节已构建图谱中因果逻辑的精准提取。该路径的核心在于将用户的自然语言指令实时转化为图数据库可执行的查询语句，其处理流程包含以下两个关键阶段。
(1) TextToCypher映射与语句动态生成
平台设计了基于指令驱动的Cypher语句生成流水线。平台通过识别提问中的实体锚点，依据5.1节定义的Schema（如设备、现象、根因等实体类型），动态拼装具备拓扑约束的查询指令。该过程将模糊的自然语言映射为逻辑严密的模式匹配语句，确保检索过程能精准对齐图谱中的“指示”或“缓解”等关系谓词。
(2)基于子图轨迹的多跳路径提取策略
在复杂的电气设备故障辅助决策场景下，单一节点的信息往往具有局限性。平台实现了基于图拓扑的深层路径提取算法，其检索深度 可设定，旨在提取出跨越物理维度与软件维度的完整因果链路。对于识别出的核心实体节点 ，平台搜索多跳关联节点集合 的表达如式(5-2)[96]。
	 	（5-2）
检索到的原始三元组集合将通过“图序列化器”转化为具备自然语言感知的文本描述段落，即可作为依据注入后续生成流中。这种基于确定性拓扑路径的动态检索，从底层逻辑上减少了生成式模型在处理严肃知识时的幻觉风险。
5.3.2 非结构化语义向量检索实现
与图轨迹检索的确定性路径提取不同，FAISS向量检索侧重于对用户查询意图的广域语义匹配，其查询时流程分为以下两个环节。
首先，平台将用户的自然语言查询文本送入与5.2.2节建库阶段完全一致的BAAI/bge-base-zh编码器，将其映射为768维查询向量q。保持编码模型一致是确保查询向量与索引向量处于同一语义空间的必要条件，若两者使用不同模型，余弦相似度计算将失去意义。其次，平台调用index.search(q , k)对持久化的IndexFlatIP索引执行Top-K近邻检索，默认返回相似度最高的5个文本切片的行号与相似度分数；随后通过与FAISS索引行号对齐的元数据映射表，将行号还原为原始文档片段与来源信息，最终即可输出带置信度分数的有序文本切片列表，实现非结构化语义向量检索。
相较于图轨迹检索，FAISS向量检索的优势在于对语义相近但表述不同的查询具有更强的覆盖能力（如"绕组匝间短路"与"线圈内部漏电"可被映射至相近向量），弥补了图谱只能匹配已知实体节点的局限性。
5.3.3 基于LLM的意图路由检索

(1)路由器分级设计

5.4 平台智能助手交互实现
在完成图文双轨知识检索后，平台智能体助手进入知识聚合与推理生成阶段。本节介绍平台如何将检索回的异构碎片化知识，结合平台实时运行状态进行动态拼装，并通过流式交互技术将复杂的诊断建议精准呈现给用户。
5.4.1 动态上下文隐式状态注入机制
为了使智能助手具备环境感知能力，平台设计并实现了一套动态上下文隐式状态注入机制，该机制旨在解决长对话过程中用户意图丢失的痛点，确保智能助手的回复具备针对性。

图5 4 动态上下文隐式状态注入机制流程图
如图5-4，当用户在前端发起提问时，平台会通过Vue全局状态管插件Pinia静默采集当前页面的活跃数据集元数据。这些数据涵盖了当前上传或选中的数据集唯一标识、所属的场类别（电场或磁场）、数据集包含的特征参数维度，以及最近一次执行预测任务的模型配置。前端将这些非查询类的结构化信息封装为“任务环境快照”，随用户请求一同发送至后端网关。
后端接收到环境快照后，首先执行语义对齐程序，将当前数据集的物理特征映射为5.1节定义的模式约束标签，从而将孤立的任务参数转化为具备领域语义的推理基准。随后，平台将对齐后的语义标签与实时元数据进行动态封装，并以平台变量的形式注入大语言模型的Prompt预置上下文中，将实时的“任务态”转化为模型推理的逻辑背景。这种隐式注入机制不仅补全了对话过程中的环境缺省，也为下一节中异构知识的精确重排与融合提供了必要的上下文参照依据。
5.4.2 基于知识双轨融合的提示词编排与重排
在获取检索结果与环境快照后，平台进入动态上下文拼装阶段。由于图谱三元组与向量文本的语义密度不同，平台设计并实现了一套基于优先级权重的上下文重排策略，如图5-5为该策略的核心重排流程。该策略旨在实现多模态工程知识与实时任务流的高保真聚合，确保最终生成的提示词有严密的逻辑层级。
首先，平台对Neo4j返回的原始子图执行序列化转换，将其映射为具备自然语言感知的逻辑事实描述。这些源自5.1节模式约束的三元组被提取出来，并置于提示词Prompt的高权重头部作为“逻辑锚点”。这种强制性的前置编排方式，使得LLM必须优先基于确定的物理机理进行因果推演，从底层逻辑上消减了生成式模型在工业严肃场景下容易产生的知识幻觉风险。
其次，平台将量库检索召回的Top-K文本切片作为背景补充，顺序编排在提示词的后续区域，用以提供详细的操作规程与数值指导。
接着，在多模态数据的物理缝合阶段，为了避免繁琐且易错的字符串硬编码，平台底层引入了LangChain编排框架中的动态提示词模板组件。如图5-5中部的“动态上下文拼装”模块，该组件充当了系统的数据总线。它通过定义严密的变量占位符，将前文注入的平台环境背景、前置的图谱逻辑锚点、向量背景补充及用户的原始提问进行结构化对齐，最终缝合出完整的增强提示词，确保系统生成的辅助决策报告具备严谨的因果推演逻辑。
最后，考虑到数据规范问题，平台调用的是云端Qwen大模型，平台在提示词送入LLM前内置了轻量级的数据脱敏规则，利用正则表达式，自动将提示词中的敏感标签进行匿名化替换，保障了数据安全性。

5.5 智能助手问答实验与结果分析
前文介绍了KG-RAG智能助手的"建库、检索、交互"全链路实现。为量化验证该助手系统相对于传统问答方案的在本平台的实际应用效果，本节设计并执行了对比实验，从检索质量与生成质量两个层次对系统性能进行综合评估。评估框架基本参照检索增强生成领域广泛认可的RAGAS（Retrieval Augmented Generation Assessment）评估体系[97]。

5.5.3 实验设计与结果分析
首先，通过不同的基线方法，来对比本文设计的基于KG-RAG的平台助手的问答效果，具体实验设计如下。
基线1（原生LLM）：直接将用户问题送入Qwen API，不附加任何检索增强上下文，以此衡量原生大模型在垂直领域的基础能力上限。
基线2（传统RAG）：仅启用FAISS向量库进行语义检索，召回文本切片作为上下文后调用Qwen API生成答案，代表传统RAG范式。
本文方法（KG-RAG）：启用本文融合图谱的双轨混合检索架构KG-RAG，包括Neo4j图谱轨迹检索、FAISS向量检索、意图路由器及并发调度模块，充分利用本文全部设计。需要说明的是，5.4节所述的动态上下文隐式状态注入机制属于平台生产环境中面向真实用户会话的UI层功能，其注入内容因用户实际操作状态动态变化，不具备受控实验的可复现性，故本实验将该机制排除在对比范围之外，以验证本文智能助手架构的工程有效性。
接着，为确保实验公平性与结果可复现性，本实验对涉及生成与评估的公共环节实施了严格的变量控制。在生成阶段，三组方法均统一调用相同版本的qwen-plus作为基础模型，并将其温度系数（Temperature）严格固定为0.1以降低自回归生成随机性带来的偏差，最大生成长度（Max_tokens）设定为1024；在检索阶段，传统RAG与KG-RAG强制对齐语义基座，其中FAISS向量召回环节均设定一致的召回数量上限Top-K=5，确保注入的上下文信息增量处于同一量级；在评估阶段，RAGAS管线全局固定采用qwen-plus作为裁判模型，并统一采用完全贪婪解码，保障LLM-as-Judge自动化打分客观性。
然后，基于RAGAS自动化评估框架，针对51道题目，三组方法分别独立运行并记录（问题、检索上下文、生成答案）三元组。

5.6 本章小结
本章围绕平台智能化交互需求，按照"数据底座、架构调度、应用落地、效果验证"的完整闭环，详细论述了基于KG-RAG混合检索增强架构的智能中枢设计与实现。
首先，在数据底座层面，创新性地设计并构建了面向设备运维与算法排错的双维度知识图谱，结合LLM少样本指令工程实现了高可靠的自动化三元组抽取，并依托Neo4j图数据库完成了结构化实体节点与关系边的入库存储；同时构建了基于FAISS的轻量级向量数据库，通过滑动窗口切片与中文稠密检索模型完成了11份工程文档共107条切片的持久化语义索引，两库共同奠定了双轨知识底座。
其次，在架构调度层面，设计了融合图轨迹检索、FAISS语义检索与分级意图路由的混合检索增强架构，通过异步并发调度实现了双轨引擎的高效协同，并构建了多级降级容错机制，保障了平台高可用性。
然后，在应用落地层面，依托LangChain编排框架与Pinia状态管理，实现了动态上下文隐式注入、优先级加权的提示词重排以及基于SSE协议的前端流式响应，打通了从底层知识检索到前端归因交互的全流程智能闭环。
最后，在效果验证层面，依托双轨知识底座，采用RAGAS框架对原生LLM、传统RAG与本文KG-RAG进行了自动化对比评估。结果表明，KG-RAG在上下文召回率（CR）、忠实度（F）与答案相关性（AR）三大核心指标上全面领先。特别是在设备故障诊断场景中，该架构有效抑制了生成幻觉并保障了因果归因的完整性，验证了双轨混合检索在工业垂直领域的有效性与工程落地价值。
